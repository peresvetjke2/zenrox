# PRD-001: First Capture And Retrieval Loop

## Problem

`zenrox` уже имеет общий product context на уровне [../domain/problem.md](../domain/problem.md), но пока не зафиксирована первая конкретная инициатива, которая превращает этот контекст в минимально полезный пользовательский результат.

Сейчас у пользователя нет надежного короткого цикла: завести личную задачу, сохранить ее в системе и позже получить полный список незавершенных задач без ручного перебора заметок, чатов или внешних списков. Без этого `zenrox` остается скорее идеей personal assistant, чем рабочей внешней памятью для повседневной жизни.

Первая инициатива должна закрыть именно этот разрыв. Ее задача не в том, чтобы покрыть весь vision продукта, а в том, чтобы доказать на реальном single-user usage, что `zenrox` уже умеет хранить открытые задачи и возвращать их пользователю в полезном виде. Более сложный extraction из свободного текста может стать отдельным следующим слоем и не должен размывать первый MVP.

## Users And Jobs

| User / Segment | Job To Be Done | Current Pain |
| --- | --- | --- |
| `primary-user` | Быстро завести личную задачу и быть уверенным, что она попала в систему | Без простого контура фиксации пользователь продолжает держать задачи в голове или в разрозненных инструментах |
| `primary-user` | Позже получить полный список всех открытых задач без фильтрации по датам и без ручного перебора истории | Задачи без срока особенно легко теряются, если система умеет отвечать только на date-based вопросы |
| `primary-user` | Понять текущее состояние личного бэклога в одном ответе | Накопленные задачи не превращаются автоматически в обзор открытых обязательств |

## Goals

- `G-01` Дать пользователю возможность завести одну личную задачу одной короткой текстовой репликой без обязательного предварительного структурирования.
- `G-02` Поддержать первый retrieval-сценарий по ранее сохраненным данным: ответ на вопрос `какие у меня открытые дела?`.
- `G-03` Дать пользователю единый обзор открытых задач, включая дела без срока выполнения.
- `G-04` Доказать, что первый полезный task loop работает на мобильном single-user usage без тяжелого setup и без дополнительных операторских действий.

## Non-Goals

- `NG-01` Напоминания, proactive notifications и любая timed delivery логика не входят в эту инициативу.
- `NG-02` Голосовой ввод не является частью первого PRD; достаточно текстового входа.
- `NG-03` Проекты, подзадачи, сложная иерархия task-объектов и rich linking между сущностями не входят в первый scope.
- `NG-04` Инициатива не должна неявно превращаться в multi-user, collaborative или permissions-aware продукт.
- `NG-05` В рамках этого PRD не требуется закрывать все возможные типы retrieval; достаточно ограниченного вопроса про открытые дела.
- `NG-06` Глубокий extraction из свободного текста, включая разбор нескольких сущностей, дат, сумм и сложных атрибутов, не является целью этой инициативы и должен рассматриваться отдельно.
- `NG-07` Явное управление lifecycle задачи, включая пользовательскую команду перевода `task` в `done`, не входит в первый MVP и должно оформляться отдельным downstream feature package.

## Product Scope

Инициатива описывает первый минимальный capability-layer для `zenrox`: сохранение одной личной задачи из текстовой реплики и retrieval списка открытых задач.

### In Scope

- Пользователь может завести одну `task` из одной текстовой реплики и сохранить ее как открытое дело.
- У задачи есть минимально достаточное состояние для различения как минимум двух пользовательски значимых состояний: `open` и `done`.
- Пользователь может задать вопрос `какие у меня открытые дела?` и получить список ранее сохраненных задач, которые считаются открытыми по правилам этого PRD.
- Ответ должен включать задачи без срока выполнения, если они остаются открытыми.
- Пользовательский сценарий должен оставаться пригодным для использования с телефона.

### Out Of Scope

- Напоминания, отложенная доставка сообщений и scheduler-driven behavior.
- Проектные деревья, подзадачи, dependency graphs и другие сложные task relationships.
- Автоматическое связывание сущностей на уровне knowledge graph.
- Поддержка `note` как отдельного retrieval surface в первом MVP.
- Поддержка файлов, изображений, голосовых сообщений или других нетекстовых входов.
- Полноценный разбор нескольких независимых задач, заметок или других сущностей из одной пользовательской реплики.
- Явные пользовательские команды изменения статуса задачи, включая перевод в `done`.
- Date-based daily planning views вроде `что у меня на сегодня?` как основной retrieval-контур.
- Сложные аналитические или многошаговые вычисления по данным.
- Админские функции, shared inbox, роли, multi-user isolation.

## UX / Business Rules

- `BR-01` Первый retrieval для этой инициативы — вопрос `какие у меня открытые дела?`; он должен возвращать полный список открытых задач, а не только задачи с датой.
- `BR-02` Задачи без срока выполнения не должны исчезать из пользовательского обзора только потому, что у них нет даты.
- `BR-03` Scope этой инициативы ограничен single-user поведением и не должен порождать downstream-требования на account model или collaboration.
- `BR-04` Первый PRD должен оставаться solution-lean: downstream feature packages могут выбирать конкретный клиентский канал и внутреннюю реализацию, пока сохраняют заданный пользовательский контур.
- `BR-05` Более сложный extraction из свободного текста не должен неявно затаскиваться в эту инициативу под видом "удобства", если он расширяет продуктовый scope.
- `BR-06` Для этого PRD `open task` означает задачу, которую пользователь еще не отметил завершенной; retrieval открытых дел не должен зависеть от наличия даты, проекта или дополнительных атрибутов.
- `BR-07` Минимальный capture-контур должен оставаться user-visible простым: после сохранения assistant обязан дать короткое подтверждение, что задача принята в систему.
- `BR-08` Если одна текстовая реплика неоднозначна или содержит несколько независимых намерений, первый MVP может не пытаться разобрать ее полностью; он не должен молча создавать несколько задач без явного продуктового решения downstream.

## Success Metrics

| Metric ID | Metric | Baseline | Target | Measurement method |
| --- | --- | --- | --- | --- |
| `MET-01` | Простота фиксации новой задачи | Без `zenrox` пользователь держит задачу в голове или раскладывает ее по разным инструментам | В типовом сценарии новую задачу можно завести одной текстовой репликой и получить короткое подтверждение сохранения | Ручная сценарная проверка на реальных пользовательских примерах |
| `MET-02` | Полнота обзора открытых дел | Без системы или при date-based ответах часть задач без срока теряется из поля зрения | Вопрос `какие у меня открытые дела?` возвращает все сохраненные задачи, которые еще не отмечены завершенными, включая задачи без срока выполнения | Сценарная проверка на заранее подготовленном наборе задач со смешанными статусами и датами |
| `MET-03` | Практическая полезность первого retrieval | Ранее сохраненные данные не дают быстрый прикладной обзор текущих обязательств | Ответ на вопрос `какие у меня открытые дела?` позволяет понять текущий личный бэклог без ручного перечитывания истории и без дополнительных уточняющих фильтров | Сценарная проверка на заранее подготовленных вопросах |
| `MET-04` | Реальная пригодность для мобильного повседневного usage | Полезность assistant падает, если контур удобно использовать только с desktop | Основной task loop доступен с телефона без специального операторского обхода | Ручная проверка на мобильном устройстве |

## Risks And Open Questions

- `RISK-01` Слишком широкий первый scope может смешать product validation с задачами reminders, voice input, entity linking и сложной task-моделью, из-за чего MVP потеряет фокус.
- `RISK-02` Если downstream-реализация нарушит правило `open = not done`, retrieval может стать нестабильным и непредсказуемым для пользователя.
- `RISK-03` Инициатива может снова расплыться, если под retrieval открытых дел неявно подтянуть note retrieval, общий поиск или сложный extraction.
- `RISK-04` Даже в узком scope мобильный сценарий может деградировать, если downstream выберет клиентский канал с лишними шагами подтверждения или desktop-only ограничениями.

## Downstream Features

| Feature | Why it exists | Status |
| --- | --- | --- |
| `FT-001` | Реализует минимальный capture slice этой инициативы: одна короткая текстовая реплика превращается в одну сохраненную `open`-задачу с коротким user-visible подтверждением. В этот же package входят rejection rules для неоднозначного или multi-intent ввода и запрет на ложное подтверждение при неуспешном сохранении, потому что это часть trust-контракта именно capture-path, а не отдельная downstream capability. | `planned` |
| `FT-002` | Реализует первый retrieval slice: на вопрос `какие у меня открытые дела?` система возвращает полный список открытых задач, исключает `done` и не теряет задачи без срока выполнения. Этот package закрепляет product-инвариант `open = not done` для user-visible retrieval и подтверждает практическую полезность накопленной task-памяти. | `planned` |
