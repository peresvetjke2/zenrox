# Architecture Patterns

Этот документ задает каноничные архитектурные границы проекта `zenrox`. Он описывает не выбранный фреймворк, а устойчивые ownership-границы, integration boundaries и правила обработки данных, которые должны сохраняться при смене конкретной реализации.

Так как проект находится на ранней стадии, ниже зафиксирована минимально достаточная архитектура для personal single-user assistant. Если позже появятся multi-user requirements, offline-first режим, чувствительные данные или несколько клиентских поверхностей, этот документ нужно будет пересмотреть явно, а не размывать локальными исключениями.

## Module Boundaries

| Context | Owns | Must not depend on directly |
| --- | --- | --- |
| `interaction` | прием пользовательских сообщений, voice/text input, ответ пользователю, короткий отчет о выполненных изменениях | внутренние детали persistence schema и прямые вызовы storage |
| `assistant-orchestration` | классификация намерения, выбор сценария, вызов AI API, преобразование model output в domain commands | UI-детали клиента, SQL/DB-детали, прямое форматирование storage records |
| `personal-memory` | каноничное хранение задач, заметок, фактов и истории изменений | transport-слой клиента, prompt-логику AI orchestration |
| `retrieval-and-reasoning` | выбор релевантного контекста для ответа, подготовка данных для AI или локального расчета, доменные вычисления | прямую запись в UI, обход orchestration, произвольный доступ к чужим внутренним таблицам |
| `platform` | интеграции, конфигурация, observability, delivery/runtime plumbing | product-specific assumptions о типах пользовательских данных |

Минимальные правила для всех контекстов:

- модуль владеет своим state и публичными контрактами;
- межмодульные зависимости проходят через явно названный API, event или adapter;
- UI, jobs и интеграции не должны читать чужие внутренние детали в обход owner-модуля.
- `assistant-orchestration` не должно становиться местом долговременного хранения фактов; его задача — интерпретировать вход и выпускать команды в owner-модули.
- `personal-memory` является source of truth для сохраненных пользовательских данных; LLM output не считается сохраненным фактом до тех пор, пока не прошел через этот контекст.
- `retrieval-and-reasoning` может подготавливать данные для ответа и вычислений, но не должен подменять собой owner-логику хранения или напрямую переписывать пользовательские записи.

## Canonical Data Shapes

На текущем этапе проекту полезно мыслить не произвольными "сообщениями", а несколькими типами устойчивых записей:

- `task` — отдельное дело с возможными атрибутами: текст, due date/time, сумма, статус, источник и, когда это естественно следует из входа, пользовательский контекст вроде "работа", "личное" или другой category hint, достаточный для последующего retrieval. Assistant должен по умолчанию пытаться вывести этот контекст сам, а при недостаточной уверенности переходить в clarification вместо случайного присвоения.
- `note` — свободная заметка или фрагмент личного контекста.
- `fact` — структурированная запись, которая может описывать как долгоживущий факт, так и наблюдаемое событие во времени, и участвовать в последующих расчетах. Примеры: интервал ТО каждые 5000 км, текущий пробег 23000.
- `source message` — исходная пользовательская реплика, из которой система извлекла команды или контекст.
- `change-log entry` — внутренняя короткая запись о том, что система добавила, обновила, интерпретировала или пересчитала.

Допускается, что на раннем этапе часть этих сущностей будет храниться в одной физической схеме. Важно не физическое разделение таблиц, а логическое разделение owner-ролей и контрактов.

Дополнительные правила:

- user-facing отчет об изменениях и внутренний `change-log entry` — не одно и то же; первый оптимизируется под понятность пользователю, второй — под auditability и трассировку.
- исправления и обновления ранее сохраненных записей должны по возможности сохранять историю происхождения: из какой реплики возникла запись и какой операцией она была изменена.
- на раннем этапе проект не обязан разводить `fact` и `event` в разные доменные семейства, если это не дает практической пользы в retrieval, расчетах или UX.

## Canonical Request Flow

Основной write-flow должен выглядеть так:

1. `interaction` принимает текст или голос и нормализует вход в единый message payload.
2. `assistant-orchestration` определяет намерение: сохранить, обновить, спросить, пересчитать, уточнить.
3. `assistant-orchestration` при необходимости вызывает внешний AI API для extraction или interpretation.
4. Результат model call преобразуется в явные domain commands, а не пишется в storage "как есть".
5. `personal-memory` применяет команды, фиксирует изменения и возвращает каноничный результат операции.
6. `interaction` показывает пользователю короткий отчет: что было добавлено, изменено или понято.

Если интерпретация неуверенная, write-flow должен завершаться явным clarification verdict без сохранения данных. В таком случае система должна честно показать, что именно осталось неоднозначным, и не притворяться успешным сохранением.

Основной read-flow должен выглядеть так:

1. `interaction` принимает вопрос пользователя.
2. `assistant-orchestration` определяет, достаточно ли локального ответа или нужен retrieval.
3. `retrieval-and-reasoning` поднимает релевантный контекст из `personal-memory`.
4. Если нужен AI, в модель отправляется только достаточный для ответа контекст, а не весь storage целиком.
5. Пользователь получает ответ, при необходимости со ссылкой на использованные факты или вычисления.

Если ответ естественно сводится к простому вычислению по уже структурированным данным, предпочтителен детерминированный путь. Для остальных случаев допустим LLM-assisted reasoning. Модель должна использоваться там, где действительно нужна интерпретация, суммаризация или работа с неоднозначным естественным языком.

## Concurrency And Critical Sections

Текущий продукт single-user, но concurrency все равно возможна: мобильный клиент, повторная отправка сообщения, параллельная обработка voice/text и фоновые пересчеты.

Каноничные правила:

- write-операции над одной пользовательской командой должны иметь `operation_id` или эквивалентный idempotency key.
- повторная обработка одного и того же ввода не должна создавать дубликаты задач, заметок или фактов без явного основания.
- внешние AI-вызовы находятся вне критической секции хранения; сначала выполняется interpretation, затем атомарное применение domain commands.
- если система использует job queue, job должен быть безопасен к повторному запуску и опираться на idempotent apply semantics.
- запрещено смешивать "вызов модели" и "финальное сохранение" в одной неразличимой операции без возможности понять, была ли запись уже применена.

Idempotent recovery означает:

- повторный запуск после сетевой ошибки не должен молча дублировать данные;
- если сохранение уже произошло, система должна уметь вернуть ранее зафиксированный результат или распознать duplicate apply;
- change log должен отражать итоговое примененное состояние, а не каждый технический retry.

## Failure Handling And Error Tracking

Проект должен различать три класса неуспеха:

- `interpretation failure` — assistant не уверен, что правильно понял ввод;
- `integration failure` — неуспешный вызов AI API, speech-to-text или другой внешней зависимости;
- `persistence failure` — не удалось надежно сохранить или обновить данные.

Каноничные правила:

- `interpretation failure` переводится в user-facing verdict с просьбой переформулировать или уточнить запрос, а не в silent failure.
- `integration failure` не должен маскироваться под успешное сохранение; пользователь должен получить честный ответ, что действие не завершено.
- `persistence failure` после успешной интерпретации считается более серьезным, чем непонимание ввода, потому что создает риск расхождения между ожиданием пользователя и фактическим состоянием памяти.
- в error tracking и logs нужно передавать metadata уровня: operation id, intent type, source surface, был ли вызван внешний AI, какие domain commands предполагались к применению.
- пользовательский отчет "что было изменено" должен формироваться только после успешного применения изменений в owner-слое хранения.
- если система не уверена в интерпретации запроса, она должна предпочесть clarification вместо частичного или рискованного автосохранения.

Запрещенный паттерн: отвечать пользователю в стиле "добавил задачу" до того, как `personal-memory` подтвердил запись.

## Configuration Ownership

Так как `ops/*` еще не адаптированы под проект, здесь фиксируется только ownership-модель, а не конкретный список env vars.

Каноничные правила:

- должен существовать один schema-owner для runtime-конфигурации приложения: AI provider credentials, storage config, mobile/web integration settings, logging level, feature flags.
- defaults должны жить рядом с owner-слоем конфигурации, а не размазываться по коду в разных модулях.
- `assistant-orchestration` не должен читать сырые env vars напрямую; он должен получать уже валидированную конфигурацию через platform/config слой.
- любые изменения внешних integration contracts сначала отражаются в config owner, затем в [`../ops/config.md`](../ops/config.md), когда этот документ будет адаптирован.
- так как проект early-stage и cost-sensitive, отдельно должны быть видимы лимиты и настройки, влияющие на стоимость внешнего AI API.

## Open Assumptions

На момент первой адаптации секции `domain` еще не зафиксированы и потребуют уточнения позже:

- точный стек backend и способ хранения данных;
- формат мобильного интерфейса: web app, bot-like surface или другой клиент;
- нужна ли отдельная векторная индексация, или сначала достаточно простого retrieval по структурированным записям;
- насколько широкий класс расчетов можно и нужно закрывать локально, без обращения к LLM.
