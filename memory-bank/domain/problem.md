# Project Problem Statement

Этот документ фиксирует общий продуктовый контекст проекта `zenrox`. Feature-документы должны ссылаться на него, а не переписывать один и тот же общий контекст каждый раз.

PRD, если он нужен, не заменяет этот документ, а уточняет отдельную инициативу относительно уже зафиксированного контекста всего проекта.

## Product Context

`zenrox` — личный AI-помощник для повседневного планирования и ведения личной базы знаний. Основной интерфейс продукта — естественный диалог: пользователь формулирует мысль в свободной форме, а система должна понять намерение, выделить значимые сущности и сохранить результат в форме, пригодной для дальнейшего использования.

Первичный сценарий ориентирован на одного основного пользователя — автора проекта. Поэтому здесь важнее скорость итераций, простота эксплуатации и низкая стоимость, чем поддержка нескольких пользователей, сложная ролевая модель или тяжелые процессы сопровождения. Продукт должен быть полезен в реальной бытовой жизни: быстро принять мысль "на бегу", не потерять контекст и позже уметь достать или вычислить нужный ответ по ранее сохраненным данным.

Текущее неудовлетворенное состояние — разрозненные заметки, задачи и устные напоминания плохо превращаются в надежную внешнюю память. Обычные заметочники требуют ручной структуры, а неструктурированный чат без памяти и извлечения не дает устойчивой пользы. `zenrox` должен закрыть этот разрыв между свободным вводом и практически применимой персональной памятью.

Граница продукта на текущем этапе — персональный single-user assistant с доступом с телефона и опорой на внешний AI API. На этом уровне не предполагаются публичная платформа, командная работа или сложная экосистема расширений, если это не появится как отдельное осознанное направление позже. Для ранних итераций допустим любой простой мобильный интерфейс, если он реально позволяет использовать ключевые сценарии с телефона.

## Core Workflows

- `WF-01` Захват дел из свободного текста или голоса: пользователь отправляет одну реплику с несколькими намерениями, а помощник выделяет отдельные дела, даты, суммы и другие важные атрибуты, по возможности сам выводя пользовательский контекст вроде "работа" или "личное".
- `WF-01A` После каждого изменения assistant отчитывается коротким сообщением, что именно было добавлено, обновлено или интерпретировано, чтобы пользователь мог сразу внести корректировки следующей репликой.
- `WF-01B` Управление жизненным циклом дела: пользователь может отмечать задачу выполненной, возвращать ее в работу или иным образом менять ее статус через естественный диалог.
- `WF-01C` Иерархия дел: пользователь может вести проекты, задачи и подзадачи, а assistant должен сохранять и использовать эти связи.
- `WF-02` Ведение личных заметок и фактов: пользователь сохраняет долговременный контекст в естественной форме без жесткого шаблона, чтобы позже к нему можно было вернуться.
- `WF-02A` Связывание сущностей: пользователь может связывать заметки, задачи, проекты и факты между собой, а assistant должен учитывать эти ссылки при последующем retrieval.
- `WF-03` Извлечение и вычисление по накопленным данным: помощник должен не просто искать записи, а уметь просматривать релевантный контекст по задачам, заметкам и фактам, делать прикладной расчет и отдавать осмысленный ответ на свободные вопросы вроде "что у меня на этой неделе?" или "что мне нужно сделать по работе сегодня?".
- `WF-03A` Связанный retrieval: по запросу assistant должен уметь поднимать не только прямое совпадение, но и связанные задачи, заметки, проекты и факты, если это улучшает релевантность ответа.
- `WF-04` Мобильный доступ к ключевым сценариям: захват данных и получение ответов должны быть доступны с телефона без тяжелой настройки и без привязки к desktop-only usage.
- `WF-04A` Напоминания и proactive нотификации: пользователь может задавать напоминания, а система должна уметь доставлять их через основной клиентский интерфейс.

## Outcomes

| Metric ID | Metric | Baseline | Target | Measurement method |
| --- | --- | --- | --- | --- |
| `MET-01` | Скорость фиксации мысли, задачи или заметки | Без системы пользователь вынужден держать контекст в голове или вручную раскладывать его по разным инструментам | Идея, поручение или заметку можно сохранить одной короткой репликой без дополнительного структурирования | Ручная проверка на реальных сценариях использования |
| `MET-02` | Качество разбора естественного ввода | Свободный текст и голос не превращаются надежно в набор полезных данных | Помощник в большинстве типовых бытовых сценариев извлекает основные сущности: дела, даты, суммы, пробег, интервалы обслуживания | Ручная оценка на наборе реальных примеров пользователя |
| `MET-03` | Практическая полезность накопленных заметок | Исторические заметки сложно использовать как "внешнюю память" | По ранее сохраненным записям можно получить конкретный ответ, краткую сводку или расчет без ручного перечитывания всех заметок | Сценарная проверка на вопросах по личным данным и истории |
| `MET-04` | Доступность в реальной жизни | Полезность снижается, если доступ к assistant ограничен desktop-only usage | Ключевые сценарии захвата данных и получения ответов доступны с телефона | Ручная проверка на мобильном устройстве |
| `MET-05` | Управляемость дел | Без системы легко потерять состояние задачи, ее вложенность или связь с другими сущностями | Пользователь может отмечать выполнение дел, работать с проектами и подзадачами и получать связанный контекст без ручного bookkeeping | Сценарная проверка на реальных task flows |
| `MET-06` | Полезность напоминаний | Внешняя память теряет ценность, если сама не возвращает пользователя к важному действию вовремя | Пользователь может задать reminder и получить его в основном клиентском интерфейсе в нужный момент | Ручная проверка на тестовых напоминаниях |

## Constraints

- `PCON-01` Продукт проектируется в первую очередь для single-user usage, поэтому решения оптимизируются под личный workflow, а не под multi-tenant или collaborative scenarios.
- `PCON-02` Эксплуатационная стоимость должна оставаться минимальной; предпочтителен бесплатный или почти бесплатный хостинг, если это не ломает основной пользовательский сценарий.
- `PCON-03` Система должна быть полезна с телефона; desktop-only UX не соответствует базовой цели продукта. Для ранних релизов допустим любой простой mobile-friendly интерфейс без жесткой привязки к конкретной клиентской платформе.
- `PCON-04` На текущем этапе допустима зависимость от внешнего AI API вместо собственной модели или self-hosted inference.
- `PCON-05` Естественный ввод является primary interaction mode; свободный текст обязателен, голосовой ввод желателен, но не является обязательным для первого полезного релиза.
- `PCON-06` На текущем этапе допустима отправка личных, но не чувствительных заметок и бытовых данных во внешний AI API. Более строгая privacy-модель может быть добавлена позже как отдельное направление.
- `PCON-07` Система не должна требовать обязательного подтверждения перед каждым сохранением, но после каждого изменения должна давать короткий понятный отчет, чтобы пользователь мог быстро запросить исправление.
- `PCON-08` Основной клиент должен поддерживать доставку user-visible reminder notifications, иначе сценарий напоминаний не считается закрытым.
