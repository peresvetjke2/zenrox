# Frontend

Этот документ описывает каноничные UI-правила проекта `zenrox` на раннем этапе. Продукт строится вокруг conversational UX, поэтому главная цель frontend-слоя — не демонстрация сложного интерфейса, а быстрый и надежный доступ к ключевым сценариям capture, retrieval и clarification с телефона.

На текущем этапе предпочтительная первая surface — Telegram или другой bot-like интерфейс. Ниже зафиксированы устойчивые product-level expectations, которые этот клиент должен соблюдать. Если позднее проект выберет дополнительные поверхности, они должны наследовать эти правила, а не переопределять их неявно.

## UI Surfaces

На раннем этапе проект допускает один основной пользовательский интерфейс, если он закрывает реальные сценарии использования с телефона.

- `primary conversational surface` — Telegram или другой bot-like интерфейс для ввода сообщений, просмотра короткого отчета об изменениях и получения ответов по уже сохраненному контексту.
- `mobile-friendly access` — обязательное свойство primary surface; ключевые сценарии должны быть удобны на телефоне без desktop-only assumptions.
- `voice input` — желательная дополнительная поверхность или capability, но не обязательный блокер для первого полезного релиза.

Текущие каноничные правила:

- первая полезная версия не обязана иметь несколько клиентских приложений;
- на первом этапе одного чата достаточно; отдельный history/browse screen не является обязательной частью продукта;
- если появляется второй интерфейс, он не должен вводить отдельную product logic для записи и извлечения данных в обход общих backend-контрактов;
- canonical owner design decisions находится в domain/product layer, а не в случайных UI-экспериментах.

## Component And Styling Rules

Проекту не нужна преждевременная design system ради самой design system. На раннем этапе допустим минимальный набор UI-компонентов, если он создает последовательный и предсказуемый conversational UX.

Каноничные правила:

- UI должен быть функционально минималистичным: основное внимание на поле ввода, истории диалога, коротком текстовом отчете о действиях и читаемом ответе.
- новые визуальные паттерны допустимы только если они улучшают скорость понимания или ввода, а не добавляют декоративную сложность.
- ad hoc UI допустим внутри ранних итераций, но не должен приводить к нескольким конкурирующим паттернам для одного и того же действия.
- если появляются повторяемые элементы, их следует выносить в shared components только после того, как они реально стабилизировались в использовании.
- status states должны быть явно различимы: `processing`, `saved`, `clarification needed`, `failed`.

Для conversational продукта особенно важно:

- отчет о сохранении или изменении должен визуально отличаться от обычного assistant reply;
- clarification prompt должен быть заметен и не теряться среди обычных сообщений;
- критичные подтверждения факта "ничего не сохранено" должны быть очевидны пользователю.
- на первом этапе текстовый отчет достаточен; rich cards, structured tiles и другой расширенный UI не являются обязательными для первого полезного релиза.

## Interaction Patterns

Основной interaction pattern проекта — conversational-first.

Каноничные правила:

- основное действие пользователя — отправить свободную реплику, а не заполнить сложную форму;
- assistant должен отвечать коротко и операционно полезно: что понял, что сохранил, что не сохранил, что требует уточнения;
- UI не должен требовать обязательного ручного структурирования данных до отправки сообщения;
- если система не уверена в интерпретации, interaction pattern должен переводить пользователя в clarification flow без побочного сохранения;
- если пользователь хочет исправить результат, follow-up сообщение должно быть естественным продолжением предыдущего действия, а не отдельным сложным edit workflow.

Допустимы вспомогательные элементы:

- компактные structured previews после разбора сообщения, если они не усложняют основной chat UX;
- быстрые action controls для исправления последнего действия;
- легкий доступ к недавним заметкам, задачам или связанному контексту.

Запрещенные UX-паттерны:

- тяжелый form-first UX как основной способ работы;
- длинные многошаговые wizard flows для типовых быстрых записей;
- смешение нескольких разных paradigms ввода без явной пользы.

Предпочтительный первый клиент — Telegram или другой bot-like интерфейс. Если позже появится web app или другой клиент, он должен сохранять тот же interaction contract, а не вводить альтернативную продуктовую логику.

## Localization

На текущем этапе обязательная многоязычность не зафиксирована.

Каноничные правила:

- первичный язык интерфейса и ответов может совпадать с основным языком пользователя;
- если позже появится i18n, UI-строки и системные статусы должны локализоваться отдельно от свободного текста модели;
- пользовательские данные, заметки и assistant replies не должны насильственно нормализоваться к одному языку без явного продуктового решения;
- если будет добавлена локализация, system-owned phrases вроде `saved`, `clarification needed` и `failed` должны иметь единый owner-слой переводов.

## Open Assumptions

На момент первой адаптации секции `domain` еще не зафиксированы и потребуют уточнения позже:

- какой конкретный клиент станет первой основной surface: mobile web, bot-like interface или другой вариант;
- когда понадобится отдельный history/browse screen помимо основного диалога;
- нужен ли voice input уже в первой полезной версии или он остается следующим этапом;
- когда текстового отчета перестанет хватать и понадобятся rich UI affordances для редактирования последнего действия.
