# UC-004: Task Status Lifecycle

## Goal

Пользователь может через естественный диалог отметить задачу выполненной, вернуть ее в работу или иным образом обновить ее статус без ручного редактирования записей.

## Primary Actor

Автор проекта.

## Trigger

Пользователь отправляет реплику вроде "стоматология сделана", "закрой задачу про оплату", "верни это в работу".

## Preconditions

- В системе уже существует задача, к которой можно применить изменение статуса.
- Система может надежно определить целевую задачу или запросить уточнение.
- persistence слой поддерживает обновление task state.

## Main Flow

1. Пользователь отправляет свободную реплику с намерением изменить статус задачи.
2. Система определяет, какая задача имеется в виду и какой новый статус требуется.
3. Если уверенность достаточна, система обновляет статус задачи.
4. Система возвращает короткий отчет о том, какой статус был изменен и для какой задачи.

## Alternate Flows / Exceptions

- `ALT-01` Если пользователь ссылается на последнюю измененную или обсуждаемую задачу, система может использовать ближайший conversational context для разрешения ссылки.
- `ALT-02` Если формулировка указывает на несколько задач, система может предложить уточнение списка кандидатов.
- `EX-01` Если система не уверена, какую именно задачу нужно обновить, она не меняет статус и уходит в clarification.
- `EX-02` Если задача уже находится в требуемом статусе, система сообщает текущее состояние вместо создания лишнего изменения.
- `EX-03` Если обновление не удалось сохранить, система не выдает ложное подтверждение статуса.

## Postconditions

- После успешного завершения статус нужной задачи обновлен и traceable в истории изменений.
- Пользователь получает понятный отчет о примененном изменении.
- После неуспешного завершения задача сохраняет прежний статус.

## Business Rules

- `BR-01` Изменение статуса задачи является domain-level действием, а не только UI-маркером.
- `BR-02` Система может использовать conversational context для разрешения ссылки на задачу, но не должна угадывать при недостаточной уверенности.
- `BR-03` Любое изменение статуса подтверждается пользователю только после успешного сохранения.
- `BR-04` Повторная команда не должна создавать дублирующих state transitions без основания.
