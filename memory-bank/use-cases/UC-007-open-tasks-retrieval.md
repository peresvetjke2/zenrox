# UC-007: Open Tasks Retrieval

## Goal

Пользователь отправляет команду `задачи` и получает полный user-visible список задач в состоянии `open`, включая задачи без срока выполнения.

## Primary Actor

Автор проекта.

## Trigger

Пользователь отправляет точный запрос `задачи` через основной conversational интерфейс.

## Preconditions

- В системе уже могут существовать задачи со статусами `open` и `done`.
- Система умеет распознать запрос как retrieval-intent первого MVP, а не как capture-команду.
- Источник истины по статусу задачи находится в owner-слое task storage.

## Main Flow

1. Пользователь отправляет точную команду `задачи`.
2. Система распознает ее как retrieval-запрос полного списка открытых дел первого MVP.
3. Система читает задачи из personal memory и выбирает только записи со статусом `open`.
4. Система формирует понятный текстовый ответ: для непустого списка он начинается со строки `Открытые задачи:` и далее перечисляет задачи по одной на строку в стабильном порядке; при отсутствии открытых задач возвращается явный empty-state verdict `Открытых задач нет.`.
5. Пользователь получает ответ без изменения сохраненных задач.

## Alternate Flows / Exceptions

- `ALT-01` Если открытых задач нет, система явно сообщает `Открытых задач нет.`.
- `ALT-02` Если среди открытых задач есть записи без срока выполнения, они все равно включаются в ответ.
- `EX-01` Если чтение task storage завершается неуспешно, система сообщает `Не удалось получить список открытых задач.` и не выдает частичный или выдуманный ответ.

## Postconditions

- После успешного завершения пользователь получает полный список текущих `open`-задач или явный empty-state ответ.
- Ни одна задача не создается, не обновляется и не переводится в другой статус только из-за retrieval-запроса.
- После неуспешного завершения пользователь понимает, что reliable ответ не был получен.

## Business Rules

- `BR-01` Для первого MVP `open task` означает любую задачу, которая еще не находится в статусе `done`.
- `BR-02` Retrieval открытых дел не зависит от наличия даты, проекта, категории или других дополнительных атрибутов.
- `BR-03` В первом MVP retrieval означает только точный запрос `задачи`; этот intent не должен маршрутизироваться в capture-path.
- `BR-04` Read-only retrieval не должен менять сохраненные данные или создавать новые task records.
